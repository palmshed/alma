# ADR 0006: Anonymous Identity

**Date:** 2026-07-06
**Status:** Accepted

## Context

Alma needs to associate conversations with a user without requiring
authentication. Every browser should see its own conversations without
login. When authentication is added later, the identity layer should
switch from anonymous UUID to authenticated user ID without changing
the storage layer.

## Decision

### Identity is an application concern, not a conversation concern

Anonymous identity is established at the Flask application level via
before-request / after-request middleware, not inside any blueprint or
service. Future features (preferences, drafts, uploads, notifications,
analytics) all benefit from `g.client_id` without reinventing identity.

### Server-issued HttpOnly cookie

The `alma_client` cookie is generated by the server on the first
request and set with:

- `HttpOnly` — not accessible from JavaScript
- `SameSite=Lax` — sent on same-site navigations
- `Secure` — enabled when the request is HTTPS (production)
- `Path=/` — available to all routes
- `Max-Age=31536000` — one year

The client never generates or controls the identity value, preventing
spoofing and manipulation.

### Owner_id is opaque

`ConversationStore(owner_id=...)` accepts any string. Today it is a
UUID from `g.client_id`. Tomorrow it can be an authenticated user ID.
The storage layer is completely indifferent to the identity scheme.

### Verification covers ownership

The verification framework includes a dedicated `identity` check that
asserts:

1. First request with no cookie → `Set-Cookie` issued with all attributes
2. Second request with cookie → cookie recognized, no re-issue
3. Conversation created under identity A → visible to identity A
4. Identity B → no access to A's conversations

## Consequences

- Identity middleware lives in `create_app()`, not in any blueprint.
- All routes have `g.client_id` available without additional wiring.
- Authentication can be added later by replacing the anonymous UUID
  with a session-based identity — no storage layer changes needed.
- Platform services consume `owner_id` but do not determine how it
  is established.
- The identity check in `alma verify` permanently protects the
  ownership model from regressions.
